# ZL NuGet 发布流水线标准接入文档

> 版本: 1.1
> 最后更新: 2026-06-26
> 维护者: ZL Team
> 仓库: `github.com/qwdingyu/deploy`（公开仓库）

---

## 一、概述

本文档描述如何将一个 .NET 项目接入 ZL 统一 NuGet 发布流水线。接入后支持：

- **Tag push 自动发布**: `git tag v2.3.0 && git push origin v2.3.0`
- **手动触发**: `gh workflow run publish.yml -f version=2.3.0`
- **Dry-run 模式**: 只构建打包，不推送
- **统一版本**: 所有项目一次指定，全部使用同一版本号
- **自动混淆**: 对标记 `obfuscate: true` 的项目执行 Obfuscar 混淆（可选）
- **Trusted Publishing**: 无需长期 API Key，通过 GitHub OIDC 获取 1 小时短期 token

---

## 二、前置条件

### 2.1 仓库要求

- 项目已推送至 GitHub（公开或私有均可）
- 仓库根目录包含 `pipeline.json`（定义所有可发布项目）

### 2.2 共享工具仓库

统一发布脚本位于公开仓库 `github.com/qwdingyu/deploy`：

```
deploy/
└── scripts/
    └── publish_all.py          ← 共享发布脚本（自动读 pipeline.json）
```

### 2.3 NuGet.org 配置

需要在 [nuget.org/account/trusted-publishing](https://www.nuget.org/account/trusted-publishing) 添加一条 Trusted Publishing 策略：

| 字段 | 值 | 说明 |
|------|-----|------|
| Repository Owner | `qwdingyu` | GitHub 账号 |
| Repository | 你的仓库名 | 如 `MyProject` |
| Workflow File | `publish.yml` | workflow 文件名（不含 .github/workflows/），注意扩展名精确匹配 |

> **注意**：私有仓库的策略有 7 天激活期，需在 7 天内成功发布一次才可永久激活

## 三、项目准备

### 3.1 确定接入模式

| 模式 | 适用场景 | 特点 |
|------|---------|------|
| **模式 A — ZL 生态项目** | 使用 ZL.PFLite/ZL.IotHub 等 ZL 内部包，需混淆 | 统一版本号，受版本门禁约束 |
| **模式 B — 独立项目** | 不依赖 ZL 内部包，独立版本管理 | 自由版本号，无混淆/CPM 同步 |

**本文档两种模式都覆盖，按需选择对应的模板。**

### 3.2 csproj 发布前检查清单

在创建 `pipeline.json` 之前，先确认每个要发布的 csproj **都包含以下属性**。缺一不可，漏了一个就会导致打包失败或包元数据不完整：

```xml
<PropertyGroup>
    <!-- 必填：包唯一标识符（默认用 AssemblyName，但建议显式写） -->
    <PackageId>MyLib.Core</PackageId>

    <!-- 必填：是否允许打包 -->
    <IsPackable>true</IsPackable>

    <!-- 建议：打开后 dotnet build 自动生成 nupkg（本地开发方便） -->
    <GeneratePackageOnBuild>true</GeneratePackageOnBuild>

    <!-- 建议：版本号。不写默认 1.0.0 -->
    <Version>1.0.0</Version>

    <!-- 建议：作者名，决定 nuget.org 上显示的发布者 -->
    <Authors>MyTeam</Authors>

    <!-- 建议：包描述，显示在 nuget.org 搜索结果中 -->
    <Description>What this package does</Description>

    <!-- 建议：开源协议，显示在 nuget.org 包页面上 -->
    <PackageLicenseExpression>MIT</PackageLicenseExpression>

    <!-- 建议：仓库地址，显示在 nuget.org 包页面上 -->
    <RepositoryUrl>https://github.com/your-org/your-repo</RepositoryUrl>
</PropertyGroup>
```

> **常见遗漏**：
> - `ZL.Print.Sentinel` 原本的 `GeneratePackageOnBuild` 是 `false` → 不打包
> - `ZL.Print.WebAgent.Core` 原本没有任何 Pack 相关属性 → `dotnet pack` 不会产生 nupkg
> - `PackageId` 不写会使用 `AssemblyName`，如果 `AssemblyName` 也没写则用项目目录名

### 3.3 创建 pipeline.json

在仓库根目录创建 `pipeline.json`，定义所有需要发布的项目：

```json
{
    "$schema": "https://raw.githubusercontent.com/qwdingyu/ZL.Pipeline/main/schemas/pipeline-schema.json",
    "version": "1.0",
    "projects": [
        {
            "name": "MyLib.Core",
            "csproj": "src/MyLib.Core/MyLib.Core.csproj",
            "obfuscate": true
        },
        {
            "name": "MyLib.Extras",
            "csproj": "src/MyLib.Extras/MyLib.Extras.csproj",
            "obfuscate": false
        }
    ],
    "nugetSource": "https://api.nuget.org/v3/index.json",
    "publishTimeout": 120,
    "dryRun": false
}
```

**字段说明**：

| 字段 | 必填 | 说明 |
|------|------|------|
| `projects[].name` | 是 | 包名（必须与 `PackageId` 一致），同时也是 nupkg 文件名 |
| `projects[].csproj` | 是 | 相对于仓库根目录的 csproj 路径 |
| `projects[].obfuscate` | 是 | 是否启用 Obfuscar 混淆。独立项目通常为 `false` |
| `nugetSource` | 否 | NuGet 源，默认 `https://api.nuget.org/v3/index.json` |
| `publishTimeout` | 否 | 超时秒数 |
| `consumers` | 否 | 消费者 CPM 同步配置（仅 ZL 生态项目需要） |

### 3.4 确认目标框架与 CI 的 SDK 版本

`publish_all.py` 会自动检测每个 csproj 的目标框架。但 CI 环境需要提前安装对应的 .NET SDK：

| 目标框架 | CI 需安装的 SDK | 说明 |
|---------|----------------|------|
| `netstandard2.0` | `.NET 8.0.x` | SDK 8.0 可构建 netstandard2.0 |
| `net8.0` | `.NET 8.0.x` | —— |
| `net10.0` | `.NET 10.0.x` | 需要额外安装 |

**如果一个仓库中同时存在 netstandard2.0 和 net10.0 的项目**（如 AtomPrint），则 publish.yml 中需要安装多个 SDK：

```yaml
- uses: actions/setup-dotnet@v4
  with:
    dotnet-version: |
      8.0.x
      10.0.x
```

### 3.5 版本号管理

**模式 A（ZL 生态）推荐：写入 Directory.Build.props**

```xml
<Project>
  <PropertyGroup>
    <Version>2.2.1</Version>
  </PropertyGroup>
</Project>
```

发布时 workflow 会用 `-p:PackageVersion=` 参数覆盖，所有包统一版本。

**模式 B（独立项目）自由管理**：每个 csproj 独立写 `<Version>`，或通过 pipeline.json 手动传入。

### 3.6 NuGet.Config 处理

仓库根目录可以有 `NuGet.Config`。CI 环境下脚本会自动生成仅含 nuget.org 的临时配置，**不需要额外处理**。

如果本地开发时需要本地 feed，可以保留：

```xml
<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <packageSources>
    <clear />
    <add key="local-feed" value="/Users/xxx/.nuget/local-feed" />
    <add key="nuget.org" value="https://api.nuget.org/v3/index.json" />
  </packageSources>
</configuration>
```

### 3.7 依赖包可用性检查

发布前确保所有 NuGet 依赖项可从 `nuget.org` 解析。

**模式 A（ZL 生态）** 需要确认上游 ZL 包已在 nuget.org 发布了正确版本：

| 包 | 最新版本 | 来源仓库 |
|----|---------|---------|
| ZL.PFLite, ZL.Tag, ZL.IotHub, ZL.IotHub.Bridges | 2.2.1 | ZL.PlcBase |
| ZL.Collections, ZL.Shared, ZL.Protocol 等 24 包 | 2.2.1 | iot-sdk |
| PlcSimulator.Core, PlcSimulator.Grpc | 2.2.1 | ZL.PlcSimulator |

**模式 B（独立项目）** 不依赖 ZL 内部包，只需确认通用第三方包（SkiaSharp、Newtonsoft.Json 等）可在 nuget.org 解析即可。

## 四、创建 publish.yml

在仓库的 `.github/workflows/` 目录下创建 `publish.yml`。

### 4.1 标准模板（模式 A — ZL 生态，含混淆）

```yaml
# ============================================================
# NuGet 发布流水线（ZL 生态版 — 含 Obfuscar 混淆）
# 触发方式: tag push v* 自动发布 / workflow_dispatch 手动触发
# 核心逻辑: deploy/scripts/publish_all.py（公开仓库 qwdingyu/deploy）
# ============================================================
name: Publish to NuGet

"on":
  push:
    tags: ['v*']
  workflow_dispatch:
    inputs:
      version:
        description: 'Package version (e.g. 2.2.1)'
        required: true
        default: '2.2.1'
      skip-obfuscate:
        description: 'Skip Obfuscar obfuscation'
        type: boolean
        default: false
      dry-run:
        description: 'Dry run (pack only, no push)'
        type: boolean
        default: false

env:
  DOTNET_NOLOGO: true

jobs:
  build-and-pack:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.version.outputs.version }}
    steps:
      - uses: actions/checkout@v4

      # 检出 deploy 仓库（公开仓库，获取发布脚本）
      - name: Checkout deploy repo (shared scripts)
        uses: actions/checkout@v4
        with:
          repository: qwdingyu/deploy
          path: .pipeline
          ref: main

      # 单 SDK 时：dotnet-version: '8.0.x'
      # 多 SDK 时（如 net10.0 + netstandard2.0 混合项目）：
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: |
            8.0.x
            10.0.x

      - name: Determine version
        id: version
        run: |
          if [ "${{ github.event_name }}" = "push" ]; then
            echo "version=${GITHUB_REF_NAME#v}" >> $GITHUB_OUTPUT
          else
            echo "version=${{ inputs.version }}" >> $GITHUB_OUTPUT
          fi

      # 安装 Obfuscar（仅需要混淆时安装）
      - name: Install Obfuscar
        if: github.event_name != 'workflow_dispatch' || inputs.skip-obfuscate != true
        run: dotnet tool install --global Obfuscar.GlobalTool --version 2.2.38

      # 核心：统一脚本处理 restore → build → pack → obfuscate
      - name: Build, pack & obfuscate
        run: |
          OBF_FLAG=""
          if [ "${{ inputs.skip-obfuscate }}" = "true" ]; then
            OBF_FLAG="--skip-obfuscate"
          fi
          python3 .pipeline/scripts/publish_all.py \
            --version ${{ steps.version.outputs.version }} \
            $OBF_FLAG

      - name: Upload packages
        uses: actions/upload-artifact@v4
        with:
          name: nuget-packages-${{ steps.version.outputs.version }}
          path: artifacts/packages/*.nupkg

  push-to-nuget:
    needs: build-and-pack
    runs-on: ubuntu-latest
    if: github.event_name == 'push' || (github.event_name == 'workflow_dispatch' && inputs.dry-run != true)
    permissions:
      id-token: write
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: nuget-packages-${{ needs.build-and-pack.outputs.version }}
          path: artifacts/packages

      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'

      - name: NuGet login (OIDC → temp API key)
        uses: NuGet/login@v1
        id: login
        with:
          user: dingyuw

      - name: Push to NuGet.org
        run: |
          for pkg in artifacts/packages/*.nupkg; do
            dotnet nuget push "$pkg" \
              --source https://api.nuget.org/v3/index.json \
              --api-key ${{ steps.login.outputs.NUGET_API_KEY }} \
              --skip-duplicate
          done
```

### 4.2 精简模板（模式 B — 独立项目，无混淆）

```yaml
# ============================================================
# NuGet 发布流水线（独立项目版 — 无混淆）
# 触发方式: tag push v* 自动发布 / workflow_dispatch 手动触发
# 核心逻辑: deploy/scripts/publish_all.py（公开仓库 qwdingyu/deploy）
# ============================================================
name: Publish to NuGet

"on":
  push:
    tags: ['v*']
  workflow_dispatch:
    inputs:
      version:
        description: 'Package version'
        required: true
        default: '1.0.0'
      dry-run:
        description: 'Dry run (pack only, no push)'
        type: boolean
        default: false

env:
  DOTNET_NOLOGO: true

jobs:
  build-and-pack:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.version.outputs.version }}
    steps:
      - uses: actions/checkout@v4

      - name: Checkout deploy repo (shared scripts)
        uses: actions/checkout@v4
        with:
          repository: qwdingyu/deploy
          path: .pipeline
          ref: main

      # 多 SDK：如 net10.0 项目需加 10.0.x
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: |
            8.0.x
            10.0.x

      - name: Determine version
        id: version
        run: |
          if [ "${{ github.event_name }}" = "push" ]; then
            echo "version=${GITHUB_REF_NAME#v}" >> $GITHUB_OUTPUT
          else
            echo "version=${{ inputs.version }}" >> $GITHUB_OUTPUT
          fi

      # 无混淆，直接 build + pack
      - name: Build & pack
        run: |
          python3 .pipeline/scripts/publish_all.py \
            --version ${{ steps.version.outputs.version }} \
            --skip-obfuscate

      - uses: actions/upload-artifact@v4
        with:
          name: nuget-packages-${{ steps.version.outputs.version }}
          path: artifacts/packages/*.nupkg

  push-to-nuget:
    needs: build-and-pack
    runs-on: ubuntu-latest
    if: github.event_name == 'push' || (github.event_name == 'workflow_dispatch' && inputs.dry-run != true)
    permissions:
      id-token: write
    steps:
      - uses: actions/download-artifact@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'

      - name: NuGet login (OIDC → temp API key)
        uses: NuGet/login@v1
        id: login
        with:
          user: dingyuw

      - name: Push to NuGet.org
        run: |
          for pkg in artifacts/packages/*.nupkg; do
            dotnet nuget push "$pkg" \
              --source https://api.nuget.org/v3/index.json \
              --api-key ${{ steps.login.outputs.NUGET_API_KEY }} \
              --skip-duplicate
          done
```

### 4.3 模板要点

| 要点 | 说明 |
|------|------|
| `"on":` 加引号 | **必须**。YAML 中 `on` 是布尔关键字，不加引号会被解析为 `True` |
| `dotnet-version: \|` 多行格式 | 同时需要 .NET 8 + .NET 10 SDK 时使用。仅 .NET 8 时用单行 |
| `permissions: id-token: write` | **必须**。Trusted Publishing 需要 OIDC 权限 |
| `user: dingyuw` | nuget.org 用户名，**不是** GitHub 用户名 |
| `--skip-obfuscate` | 无混淆版本直接传此参数永远生效 |

## 五、Obfuscator 混淆说明（仅 ZL 生态项目需要）

### 5.1 启用条件

仅对 `pipeline.json` 中标记 `"obfuscate": true` 的项目执行混淆。

### 5.2 混淆参数

```xml
<Obfuscator>
  <Var name="KeepPublicApi" value="true" />     <!-- 保留公开 API -->
  <Var name="HidePrivateApi" value="true" />    <!-- 隐藏内部实现 -->
  <Var name="HideStrings" value="true" />       <!-- 字符串加密 -->
  <Var name="UseUnicodeNames" value="true" />   <!-- Unicode 混淆名 -->
  <!-- 以下全部 false，序列化不受影响 -->
  <Var name="RenameProperties" value="false" />
  <Var name="RenameEvents" value="false" />
  <Var name="RenameFields" value="false" />
  <Var name="RenameJsonProperties" value="false" />
  <Module file="$(InPath)/YourLib.dll" />
</Obfuscator>
```

### 5.3 多 TFM 处理

对于多目标框架包（`net8.0;net10.0`）：
- 脚本默认只混淆 `net8.0` 目标
- `net10.0` 框架的 DLL 保持原始状态（GitHub Actions 暂不支持 .NET 10）
- 使用 `replace_dll_in_nupkg()` 函数在 nupkg 中只替换 `lib/net8.0/` 下的 DLL

### 6.4 理解 NU5026 警告

当仓库**没有 `.sln` 解决方案文件**时，第一次 `dotnet pack` 使用 `--no-build` 会找不到 DLL：

```
error NU5026: The file '.../bin/Release/netstandard2.0/MyLib.dll' to be packed was not found on disk.
```

**这是预期行为，不是失败**。脚本会自动检测到失败并回退到单项目构建（`dotnet build` then `dotnet pack`）。

```
✅ ZL.Print.1.0.0.nupkg（单独构建）   ← 回退成功
✅ 完成! 3/3 个包已打包               ← 最终成果
```

如果仓库有 `.sln`，不会出现此警告。

## 七、常见问题

### 7.1 YAML `on` 被解析为布尔值

**现象**: `gh workflow run` 报错 `Workflow does not have 'workflow_dispatch' trigger`。

**原因**: YAML 中 `on`、`yes`、`true` 都是布尔值 `True` 的别名。不加引号的 `on` 会被 YAML 解析器理解为布尔值。

**修复**:
```yaml
# ❌ 错误
on:
  push:

# ✅ 正确
"on":
  push:
```

### 7.2 NuGet restore 失败（本地 feed 不存在）

**现象**: `error NU1301: The local source '/Users/xxx/.nuget/local-feed' doesn't exist`。

**原因**: 仓库的 `NuGet.Config` 引用了本地路径，CI 环境没有该路径。

**修复**: 无需任何修改。`publish_all.py` 会自动生成仅含 nuget.org 的临时 `NuGet.Config`，CI 下自动覆盖。

### 7.3 secrets 不可在 if 条件中使用

**现象**: 编译 workflow 时报 `Unrecognized named-value: 'secrets'`。

**原因**: GitHub Actions 的 `if:` 条件中不能引用 `secrets` 上下文，只能在 `env:` 和 `run:` 中使用。

**修复**: 将 secrets 通过 `env:` 注入，在 shell 中判断：
```yaml
- name: Push
  env:
    API_KEY: ${{ steps.login.outputs.NUGET_API_KEY || secrets.NUGET_API_KEY }}
  run: |
    if [[ -z "$API_KEY" ]]; then
      echo "::error::No API key"
      exit 1
    fi
    dotnet nuget push ... -k "$API_KEY"
```

### 7.4 Trusted Publishing 401

**现象**:
```
Token exchange failed (HTTP 401) ... No matching trust policy owned by user 'xxx'
```

**常见原因**：
| 原因 | 修复 |
|------|------|
| 策略未配置 | 去 [nuget.org](https://www.nuget.org/account/trusted-publishing) 添加 |
| Workflow 文件名拼写错误 | 策略中的 `Workflow File` 必须精确匹配文件名。`publish.yml` ≠ `publish.yaml` |
| `user` 参数填错 | `NuGet/login@v1` 的 `user:` 填的是 **nuget.org 的用户名**，不是 GitHub 账号名 |
| 私有仓库 7 天未激活 | 策略创建后 7 天内需成功发布一次才能永久激活 |
| 策略创建者和 `user` 不匹配 | 用创建策略的那个 nuget 账号的 username |

### 7.5 Obfuscar 安装失败

**现象**: `Package obfuscar is not a .NET tool`。

**原因**: 正确的 NuGet 工具包名是 `Obfuscar.GlobalTool`，不是 `Obfuscar`。前者是 dotnet tool，后者不是。

**修复**:
```bash
dotnet tool install --global Obfuscar.GlobalTool --version 2.2.38
```

### 7.6 csproj 中的绝对路径引用

**现象**: CI 构建失败，报错找不到项目文件。

**原因**: csproj 中有硬编码的本地路径（只在你的开发机上存在）：
```xml
<ProjectReference Include="/Users/xxx/other-repo/src/SomeLib/SomeLib.csproj" />
```

**修复**: 改成 NuGet 包引用（版本由 CPM 或 csproj 管理）：
```xml
<PackageReference Include="ZL.SomeLib" Version="2.2.1" />
```

### 7.7 NU5026 不是错误

**现象**: 日志中出现 `error NU5026` 但最终显示 `✅ 完成! N/N 个包已打包`。

**原因**: 脚本尝试 `--no-build` 方式 pack 但找不到 DLL（常见于无 `.sln` 的项目），自动回退到逐个构建。

**处理**: 只要最终显示 `✅ 完成!` 且包数量正确，就不需要任何操作。

### 7.8 ZL.Pipeline.Cli 与 publish_all.py 的关系

`publish_all.py` **不是** `ZL.Pipeline.Cli` 的替代品，而是 **CI 入口**：

| 工具 | 用途 | 运行环境 |
|------|------|---------|
| `ZL.Pipeline.Cli` (zl-pipeline) | 本地开发：构建、打包、混淆、消费方 CPM 同步 | 开发者本地 |
| `publish_all.py` | CI 发布：构建、打包、混淆、输出 artifact | GitHub Actions |
| GitHub Actions `publish.yml` | CI 编排：触发、依赖缓存、NuGet login、Push | GitHub Actions |

`zl-pipeline` 仍然用于：
- `zl-pipeline publish --local --version x.x.x` → 本地构建测试
- `zl-pipeline sync-consumers` → 更新消费者 CPM 版本号
- `zl-pipeline api-compare` → API 对比
