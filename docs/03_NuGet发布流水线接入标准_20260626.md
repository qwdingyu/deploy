# ZL NuGet 发布流水线标准接入文档

> 版本: 1.0
> 最后更新: 2026-06-26
> 维护者: ZL Team
> 仓库: `github.com/qwdingyu/deploy`（公开仓库）

---

## 一、概述

本文档描述如何将一个 .NET 项目接入 ZL 统一 NuGet 发布流水线。接入后支持：

- **Tag push 自动发布**: `git tag v2.3.0 && git push origin v2.3.0`
- **手动触发**: `gh workflow run publish.yml -f version=2.3.0`
- **Dry-run 模式**: 只构建打包，不推送
- **统一版本**: 所有项目一次指定，全部使用同一版本号
- **自动混淆**: 对标记 `obfuscate: true` 的项目执行 Obfuscar 混淆
- **Trusted Publishing**: 无需长期 API Key，通过 GitHub OIDC 获取 1 小时短期 token

---

## 二、前置条件

### 2.1 仓库要求

- 项目已推送至 GitHub（公开或私有均可）
- 仓库根目录包含 `pipeline.json`（定义所有可发布项目）

### 2.2 共享工具仓库

Z.L 统一发布脚本位于公开仓库 `github.com/qwdingyu/deploy`：

```
deploy/
└── scripts/
    └── publish_all.py          ← 共享发布脚本（自动读 pipeline.json）
```

### 2.3 NuGet.org 配置

需要在 [nuget.org/account/trusted-publishing](https://www.nuget.org/account/trusted-publishing) 添加一条 Trusted Publishing 策略：

| 字段 | 值 | 说明 |
|------|-----|------|
| Repository Owner | `qwdingyu` | GitHub 账号 |
| Repository | 你的仓库名 | 如 `ZL.NewProject` |
| Workflow File | `publish.yml` | workflow 文件名（不含 .github/workflows/）|

> **注意**：私有仓库的策略有 7 天激活期，需在 7 天内成功发布一次才可永久激活

## 三、项目准备

### 3.1 创建 pipeline.json

在仓库根目录创建 `pipeline.json`，定义所有需要发布的项目：

```json
{
    "$schema": "https://raw.githubusercontent.com/usethink/ZL.Pipeline/main/schemas/pipeline-schema.json",
    "version": "1.0",
    "projects": [
        {
            "name": "MyLib.Core",
            "csproj": "src/MyLib.Core/MyLib.Core.csproj",
            "obfuscate": true
        },
        {
            "name": "MyLib.Extras",
            "csproj": "src/MyLib.Extras/MyLib.Extras.csproj",
            "obfuscate": false
        }
    ],
    "nugetSource": "https://api.nuget.org/v3/index.json",
    "publishTimeout": 120,
    "dryRun": false
}
```

**字段说明**：

| 字段 | 必填 | 说明 |
|------|------|------|
| `projects[].name` | 是 | 包名（必须与 csproj 中 PackageId 一致） |
| `projects[].csproj` | 是 | 相对于仓库根目录的 csproj 路径 |
| `projects[].obfuscate` | 是 | 是否启用 Obfuscar 混淆 |
| `nugetSource` | 否 | NuGet 源，默认 `https://api.nuget.org/v3/index.json` |
| `publishTimeout` | 否 | 超时秒数 |
| `consumers` | 否 | 消费者 CPM 同步配置（可选） |

### 3.2 统一版本号管理

**方案 A（推荐）：写入 Directory.Build.props**

```xml
<Project>
  <PropertyGroup>
    <Version>2.2.1</Version>
  </PropertyGroup>
</Project>
```

这样本地开发时默认版本也是统一的。发布时 workflow 会用 `-p:PackageVersion=` 参数覆盖。

**方案 B：由流水线注入（不写死）**

csproj 和 props 中均不设 `<Version>`，让 publish_all.py 通过 `dotnet pack -p:PackageVersion=` 注入版本号。

> 至少一种方案必须存在。如果 csproj 既无 `<Version>` 也不继承 props，dotnet pack 将使用默认版本号 `1.0.0`。

### 3.3 NuGet.Config 处理

仓库根目录可以有 `NuGet.Config`。CI 环境下脚本会自动生成仅含 nuget.org 的临时配置，**不需要额外处理**。

如果本地开发时需要本地 feed，可以保留，CI 会自动覆盖：

```xml
<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <packageSources>
    <clear />
    <add key="local-feed" value="/Users/xxx/.nuget/local-feed" />
    <add key="nuget.org" value="https://api.nuget.org/v3/index.json" />
  </packageSources>
</configuration>
```

### 3.4 依赖包可用性检查

发布前需要确保所有 NuGet 依赖项可从 `nuget.org` 解析。

如果项目依赖 ZL 内部包（如 `ZL.PFLite`、`ZL.IotHub` 等）：

1. 先确保上游仓库已发布对应版本到 nuget.org
2. 在 `Directory.Packages.props` 或 csproj 中引用正确版本号

**ZL 内部包版本对照表（当前）**：

| 包 | 最新版本 | 来源仓库 |
|----|---------|---------|
| ZL.PFLite, ZL.Tag, ZL.IotHub, ZL.IotHub.Bridges | 2.2.1 | ZL.PlcBase |
| ZL.Collections, ZL.Shared, ZL.Protocol 等 24 包 | 2.2.1 | iot-sdk |
| PlcSimulator.Core, PlcSimulator.Grpc | 2.2.1 | ZL.PlcSimulator |

## 四、创建 publish.yml

在仓库的 `.github/workflows/` 目录下创建 `publish.yml`：

### 标准模板（复制即用）

```yaml
# ============================================================
# NuGet 发布流水线
# 触发方式: tag push v* 自动发布 / workflow_dispatch 手动触发
# 核心逻辑: deploy/scripts/publish_all.py（公开仓库 qwdingyu/deploy）
# ============================================================
name: Publish to NuGet

"on":
  push:
    tags: ['v*']
  workflow_dispatch:
    inputs:
      version:
        description: 'Package version (e.g. 2.2.1)'
        required: true
        default: '2.2.1'
      skip-obfuscate:
        description: 'Skip Obfuscar obfuscation'
        type: boolean
        default: false
      dry-run:
        description: 'Dry run (pack only, no push)'
        type: boolean
        default: false

env:
  DOTNET_NOLOGO: true

jobs:
  build-and-pack:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.version.outputs.version }}
    steps:
      - uses: actions/checkout@v4

      # 检出 deploy 仓库（公开仓库，获取发布脚本）
      - name: Checkout deploy repo (shared scripts)
        uses: actions/checkout@v4
        with:
          repository: qwdingyu/deploy
          path: .pipeline
          ref: main

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'

      - name: Determine version
        id: version
        run: |
          if [ "${{ github.event_name }}" = "push" ]; then
            echo "version=${GITHUB_REF_NAME#v}" >> $GITHUB_OUTPUT
          else
            echo "version=${{ inputs.version }}" >> $GITHUB_OUTPUT
          fi

      # 安装 Obfuscar（仅需要混淆时安装）
      - name: Install Obfuscar
        if: github.event_name != 'workflow_dispatch' || inputs.skip-obfuscate != true
        run: dotnet tool install --global Obfuscar.GlobalTool --version 2.2.38

      # 核心：统一脚本处理 restore → build → pack → obfuscate
      - name: Build, pack & obfuscate
        run: |
          OBF_FLAG=""
          if [ "${{ inputs.skip-obfuscate }}" = "true" ]; then
            OBF_FLAG="--skip-obfuscate"
          fi
          python3 .pipeline/scripts/publish_all.py \
            --version ${{ steps.version.outputs.version }} \
            $OBF_FLAG

      - name: Upload packages
        uses: actions/upload-artifact@v4
        with:
          name: nuget-packages-${{ steps.version.outputs.version }}
          path: artifacts/packages/*.nupkg

  push-to-nuget:
    needs: build-and-pack
    runs-on: ubuntu-latest
    # 只在非 dry-run 时执行推送
    if: github.event_name == 'push' || (github.event_name == 'workflow_dispatch' && inputs.dry-run != true)
    permissions:
      id-token: write     # Trusted Publishing 需要
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: nuget-packages-${{ needs.build-and-pack.outputs.version }}
          path: artifacts/packages

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'

      # Trusted Publishing: 通过 GitHub OIDC 获取 1 小时短期 API Key
      - name: NuGet login (OIDC → temp API key)
        uses: NuGet/login@v1
        id: login
        with:
          user: dingyuw

      - name: Push to NuGet.org
        run: |
          for pkg in artifacts/packages/*.nupkg; do
            echo "  ➡ $(basename $pkg)"
            dotnet nuget push "$pkg" \
              --source https://api.nuget.org/v3/index.json \
              --api-key ${{ steps.login.outputs.NUGET_API_KEY }} \
              --skip-duplicate
          done
          echo "✅ Published version ${{ needs.build-and-pack.outputs.version }} to NuGet.org"
```

### 模板说明

| 部分 | 说明 |
|------|------|
| `"on":` 加引号 | **必须**。YAML 中 `on` 是布尔关键字，不加引号会被解析为 `True`，导致 workflow 无法触发 |
| `push: tags: ['v*']` | 打 tag 时自动触发，tag 名如 `v2.3.0` |
| `inputs.version` | 手动触发时指定版本号 |
| `inputs.skip-obfuscate` | 跳过混淆（快速测试用） |
| `inputs.dry-run` | 只打包不推送 |
| `permissions: id-token: write` | **必须**。Trusted Publishing 需要 OIDC 权限 |
| `user: dingyuw` | nuget.org 用户名，**不是** GitHub 用户名 |

## 五、Obfuscator 混淆说明

### 5.1 启用条件

仅对 `pipeline.json` 中标记 `"obfuscate": true` 的项目执行混淆。

### 5.2 混淆参数

```xml
<Obfuscator>
  <Var name="KeepPublicApi" value="true" />     <!-- 保留公开 API，消费者无感 -->
  <Var name="HidePrivateApi" value="true" />    <!-- 隐藏内部实现 -->
  <Var name="HideStrings" value="true" />       <!-- 字符串加密 -->
  <Var name="UseUnicodeNames" value="true" />   <!-- Unicode 混淆名 -->
  <!-- 以下全部 false，不重命名，序列化不受影响 -->
  <Var name="RenameProperties" value="false" />
  <Var name="RenameEvents" value="false" />
  <Var name="RenameFields" value="false" />
  <Var name="RenameJsonProperties" value="false" />
  <Module file="$(InPath)/YourLib.dll" />
</Obfuscator>
```

### 5.3 多 TFM 处理

对于多目标框架包（`net8.0;net10.0`）：
- 脚本默认只混淆 `net8.0` 目标
- `net10.0` 框架的 DLL 保持原始状态（GitHub Actions 暂不支持 .NET 10）
- 使用 `replace_dll_in_nupkg()` 函数在 nupkg 中只替换 `lib/net8.0/` 下的 DLL

### 5.4 已有混淆配置的兼容

如果项目已经有 `obfuscar.xxx.xml` 文件，脚本会忽略它们，使用脚本内联生成的配置（保证参数一致性）。

## 六、测试验证

### 6.1 验证 pipeline.json

确保 JSON 格式正确、所有 csproj 路径存在：

```bash
python3 -c "import json; json.load(open('pipeline.json')); print('✅')"
ls -la $(python3 -c "import json; [print(p['csproj']) for p in json.load(open('pipeline.json'))['projects']]")
```

### 6.2 本地测试构建

```bash
# 安装 Obfuscar
dotnet tool install --global Obfuscar.GlobalTool --version 2.2.38

# 运行发布脚本（本地模式）
python3 .pipeline/scripts/publish_all.py --version 2.2.1 --local
```

### 6.3 触发 CI dry-run

```bash
# 先提交并推送 workflow 文件
git add .github/workflows/publish.yml
git commit -m "feat: 添加 NuGet 发布流水线"
git push

# 触发 dry-run（只打包不推送）
gh workflow run publish.yml --ref main \
  -f version=0.0.1-test \
  -f skip-obfuscate=true \
  -f dry-run=true

# 查看运行日志
gh run watch
```

### 6.4 正式发布测试

```bash
# 先确认 dry-run 通过后，触发真实发布（推送到 nuget.org）
gh workflow run publish.yml --ref main \
  -f version=0.0.1-test \
  -f dry-run=false
```

> 测试完成后到 [nuget.org](https://www.nuget.org) 上 unlist 测试版本。

### 6.5 验证 Tag 自动发布

```bash
git tag v0.0.1-test-tag
git push origin v0.0.1-test-tag
# 自动触发 publish.yml，监控 GitHub Actions 日志
```

## 七、常见问题

### 7.1 YAML `on` 被解析为布尔值

**现象**: `gh workflow run` 报错 `Workflow does not have 'workflow_dispatch' trigger`

**原因**: YAML 中 `on` 等价于 `true`，不加引号会被解析为布尔值

**修复**: 
```yaml
# ❌ 错误
on:
  push:

# ✅ 正确
"on":
  push:
```

### 7.2 NuGet restore 失败（本地 feed 不存在）

**现象**: `error NU1301: The local source '/Users/xxx/.nuget/local-feed' doesn't exist`

**原因**: 仓库的 `NuGet.Config` 引用了本地路径，CI 环境没有该路径

**修复**: 无需任何修改。`publish_all.py` 会自动生成仅含 nuget.org 的临时 `NuGet.Config`，CI 下自动覆盖。

### 7.3 secrets 不可在 if 条件中使用

**现象**: `Unrecognized named-value: 'secrets'`

**原因**: GitHub Actions 的 `if:` 条件中不能引用 `secrets` 上下文

**修复**: 将 secrets 通过 `env:` 注入，在 shell 中判断：

```yaml
- name: Push
  env:
    NUGET_API_KEY: ${{ steps.login.outputs.NUGET_API_KEY }}
  run: |
    if [[ -z "$NUGET_API_KEY" ]]; then
      echo "::error::No API key available"
      exit 1
    fi
    dotnet nuget push ... -k "$NUGET_API_KEY"
```

### 7.4 Trusted Publishing 401 / Token exchange failed

**现象**:
```
Token exchange failed (HTTP 401) ... No matching trust policy
```

**常见原因**：
- 策略未配置 → 去 [nuget.org/account/trusted-publishing](https://www.nuget.org/account/trusted-publishing) 添加
- Workflow 文件名不匹配 → 检查策略中的 `Workflow File` 是否与 `publish.yml`（或 `publish.yaml`）完全一致
- `user` 参数错误 → `NuGet/login@v1` 的 `user:` 填的是 nuget.org 的用户名，**不是** GitHub 用户名
- 私有仓库 7 天未激活 → 需要在 7 天内成功发布一次

### 7.5 Obfuscar 安装失败

**现象**: `Package obfuscar is not a .NET tool`

**原因**: 正确的 NuGet 工具包名是 `Obfuscar.GlobalTool`（命令名 `obfuscar.console`）

**修复**:
```bash
dotnet tool install --global Obfuscar.GlobalTool --version 2.2.38
```

### 7.6 csproj 中的绝对路径引用

**现象**: CI 构建失败，报错找不到项目文件

**原因**: csproj 中有硬编码的本地路径：
```xml
<ProjectReference Include="/Users/xxx/other-repo/src/SomeLib/SomeLib.csproj" />
```

**修复**: 改成 NuGet 包引用：
```xml
<PackageReference Include="ZL.SomeLib" />
```

### 7.7 ZL.Pipeline.Cli 与 publish_all.py 的关系

`publish_all.py` **不是** `ZL.Pipeline.Cli` 的替代品，而是 **CI 入口**：

| 工具 | 用途 | 运行环境 |
|------|------|---------|
| `ZL.Pipeline.Cli` (zl-pipeline) | 本地开发：构建、打包、混淆、消费方 CPM 同步 | 开发者本地 |
| `publish_all.py` | CI 发布：构建、打包、混淆、输出 artifact | GitHub Actions |
| GitHub Actions `publish.yml` | CI 编排：触发、依赖缓存、NuGet login、Push | GitHub Actions |

`zl-pipeline` 仍然用于：
- `zl-pipeline publish --local --version x.x.x` → 本地构建测试
- `zl-pipeline sync-consumers` → 更新消费者 CPM 版本号
- `zl-pipeline api-compare` → API 对比

## 八、接入清单

一个新项目的完整接入步骤：

- [ ] 1. NuGet.org 配置 Trusted Publishing 策略
- [ ] 2. 仓库根目录创建 `pipeline.json`，列出所有发布项目
- [ ] 3. 确认或添加统一版本号（`Directory.Build.props` 或流水线注入）
- [ ] 4. 确认所有 NuGet 依赖可从 nuget.org 解析
- [ ] 5. 创建 `.github/workflows/publish.yml`（使用本文模板）
- [ ] 6. 提交推送 workflow 文件
- [ ] 7. 触发 dry-run 测试（`gh workflow run ... -f dry-run=true`）
- [ ] 8. 确认构建成功、包数量正确
- [ ] 9. 触发正式发布测试（`-f dry-run=false`，版本用 `0.0.1-test`）
- [ ] 10. 确认包已推送到 nuget.org，unlist 测试版本
- [ ] 11. 测试 tag 触发：`git tag v0.0.1-test-tag && git push origin v0.0.1-test-tag`
