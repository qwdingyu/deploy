## 1. 工具定位与使用场景

### 1.1 一句话定位

**zl-pipeline** 是一个面向 .NET 库项目的 **NuGet 正式发布流水线 CLI 工具**，集中管理 build → pack → fix-nuspec → publish-deps → obfuscate → replace-dll → api-compare → push 全链路。

### 1.2 使用场景

| 场景 | 如何使用 |
|------|---------|
| 日常本地开发快速迭代 | 用 `local-pack` 工具，不经过 zl-pipeline |
| **正式发布到 NuGet.org** | **用 `zl-pipeline publish`** |
| 正式发布前验证流程是否正确 | 用 `zl-pipeline publish --dry-run` |
| 验证已发布的 NuGet 包是否包含混淆 | 用 `zl-pipeline check <包名> <版本>` |
| 发布后同步下游消费者 CPM 版本号 | 用 `zl-pipeline sync-consumers <版本>` |
| 独立发版时对齐消费者 CPM | 用 `zl-pipeline align-versions` |

### 1.3 架构概览

```
zl_pipeline/
├── main.py              # CLI 入口，argparse 子命令解析
├── config.py            # pipeline.json 解析与验证
├── context.py           # 不可变运行上下文（PipelineContext）
├── runner.py            # StepRunner 注册表 + 执行引擎
├── result.py            # 步骤执行结果（StepResult）
├── state.py             # 断点续传状态持久化
├── report.py            # 终端彩色汇总报告
├── dotnet.py            # dotnet CLI 适配器
├── obfuscar.py          # Obfuscar 适配器
└── steps/
    ├── build.py         # 步骤 1: 编译
    ├── pack.py          # 步骤 2: NuGet 打包
    ├── fix_nuspec.py    # 步骤 3: 修复 nuspec 依赖版本
    ├── publish_deps.py  # 步骤 4: 发布依赖集
    ├── obfuscate.py     # 步骤 5: Obfuscar 混淆
    ├── replace_nupkg.py # 步骤 6: 替换 nupkg 中 DLL
    ├── api_compare.py   # 步骤 7: API 完整性对比
    └── push.py          # 步骤 8: 推送到 NuGet.org
```

### 4.1 `init` — 初始化配置

在当前目录生成 `pipeline.json`，自动扫描项目目录下所有库项目（.csproj 文件），排除测试项目、性能测试项目、示例项目和 Console 应用。

**命令格式**

```bash
zl-pipeline init
```

**无需任何参数。**

**验证结果**（2026-06-13 实际运行）：
```bash
$ uv run python -m zl_pipeline init --help
usage: zl-pipeline init [-h]

options:
  -h, --help  show this help message and exit
```

**注意事项**：
- 生成后**必须检查**生成的 `pipeline.json`，有时会自动包含不应发布的 demo/bench 项目
- 如果项目根目录不在当前目录，需要手动编辑 csproj 路径

### 4.3 `verify` — 验证模式（dry-run 发布）

执行完整流水线，但**不推送 NuGet**。实际构建、实际混淆、实际对比 API，只是最后一步推送被跳过。

**命令格式**

```bash
zl-pipeline verify <version> \
  [--only PROJECTS...] \
  [--from-step STEP_NAME] \
  [--skip-build] \
  [--resume]
```

| 参数 | 说明 |
|------|------|
| `version` | 版本号（必填） |
| `--only` | 仅处理指定项目（可多次使用） |
| `--from-step` | 从指定步骤名开始（如 `obfuscate`） |
| `--skip-build` | 跳过编译步骤 |
| `--resume` | 断点续跑（恢复上次未完成的步骤） |

**验证结果**（2026-06-13 实际运行，dry-run 模式 8/8 全部通过）：

```bash
# 测试 ZL.PlcBase 项目
$ zl-pipeline verify 1.0.0-test --only ZL.PlcBase

# 输出:
# [ZL.PlcBase] [PASS] build (dry-run 跳过)
# [ZL.PlcBase] [PASS] pack (dry-run 跳过)
# [ZL.PlcBase] [PASS] fix_nuspec (dry-run 模式，跳过修复)
# [ZL.PlcBase] [PASS] publish_deps (dry-run 跳过)
# [ZL.PlcBase] [PASS] obfuscate (dry-run 模式，跳过混淆)
# [ZL.PlcBase] [PASS] replace_nupkg (dry-run 模式，跳过替换)
# [ZL.PlcBase] [PASS] api_compare (dry-run 跳过)
# [ZL.PlcBase] [SKIP] push: dry-run 模式，跳过推送
# ===========================================
# Total: 8 passed, 0 failed, 1 skipped
# ===========================================
```

**关键说明**：verify 命令等价于 `publish --dry-run`。在 dry-run 模式下，所有需要物理文件的步骤（fix_nuspec、obfuscate、replace_nupkg）会自动跳过，返回成功。

### 4.4 `publish` — 发布模式（完整流水线）

执行完整流水线并推送到 NuGet.org。

**命令格式**

```bash
zl-pipeline publish <version> \
  [--only PROJECTS...] \
  [--from-step STEP_NAME] \
  [--skip-build] \
  [--resume]
```

参数与 `verify` 完全一致，区别在于：
- `verify` 始终 dry-run（不推送）
- `publish` 始终真实执行（会推送）

**验证结果**（2026-06-13 实际运行）：

```bash
$ uv run python -m zl_pipeline publish --help

                           [--from-step FROM_STEP] [--skip-build] [--resume]
                           version

positional arguments:
  version               版本号

options:
  -h, --help            show this help message and exit
  --only ONLY [ONLY ...]
                        仅处理指定项目
  --from-step FROM_STEP
                        从指定步骤开始
  --skip-build          跳过编译
  --resume              断点续跑
```

**发布前置条件**：

## 5. pipeline.json 配置详解

### 5.1 文件位置

`pipeline.json` 放在项目根目录（与 `Directory.Packages.props` 同级）。

### 5.2 完整配置示例

```json
{
  "version": "1.0",
  "projects": [
    {
      "name": "ZL.Watchdog",
      "csproj": "src/platform/ZL.Watchdog/ZL.Watchdog.csproj",
      "obfuscate": true,
      "obfuscarConfig": "obfuscar.xml",
      "includeDependencies": []
    },
    {
      "name": "ZL.Protocol",
      "csproj": "src/protocols/ZL.Protocol/ZL.Protocol.csproj",
      "obfuscate": false
    }
  ],
  "obfuscarConfig": "obfuscar.xml",
  "nugetSource": "https://api.nuget.org/v3/index.json",
  "publishTimeout": 120,
  "dryRun": false,
  "consumers": [
    {
      "name": "tmom",
      "path": "/Users/dingyuwang/0-X/tmom",
      "cpmFile": "Directory.Packages.props",
      "buildTarget": "api/TMom.Device.Runtime.Host/TMom.Device.Runtime.Host.csproj",
      "autoCommit": false
    }
  ]
}
```

### 5.3 字段说明

#### 顶层字段

| 字段 | 必填 | 默认值 | 说明 |
|------|------|--------|------|
| `version` | 是 | - | 配置格式版本，当前为 "1.0" |
| `projects` | 是 | - | 要发布的库项目数组，至少一个 |
| `obfuscarConfig` | 否 | `obfuscar.xml` | 全局默认 Obfuscar 配置文件路径 |
| `nugetSource` | 否 | `https://api.nuget.org/v3/index.json` | NuGet 推送源地址 |
| `publishTimeout` | 否 | `120` | 推送超时秒数 |
| `dryRun` | 否 | `false` | 全局 dry-run 模式 |
| `consumers` | 否 | `[]` | 下游消费项目列表 |

#### projects[].（单个项目配置）

| 字段 | 必填 | 默认值 | 说明 |
|------|------|--------|------|
| `name` | 是 | - | 项目名（同时也是包名/DLL 名） |
| `csproj` | 是 | - | 相对于项目根目录的 .csproj 路径 |
| `obfuscate` | 否 | `true` | 是否对此项目执行混淆 |
| `obfuscarConfig` | 否 | 全局值 | 自定义 Obfuscar 配置，覆盖全局设置 |
| `includeDependencies` | 否 | `[]` | 额外需要包含的依赖 DLL 列表 |

#### consumers[].（消费者配置）

| 字段 | 必填 | 默认值 | 说明 |
|------|------|--------|------|
| `name` | 是 | - | 消费者名称（显示用） |
| `path` | 是 | - | 消费者项目根目录绝对路径 |
| `cpmFile` | 否 | 自动查找 | CPM 文件名（默认 `Directory.Packages.props`） |
| `buildTarget` | 否 | - | 同步版本后验证编译的目标项目 |
| `autoCommit` | 否 | `false` | 是否自动 git commit + push |
| `autoDiscover` | 否 | `false` | 是否自动扫描子目录中的 CPM |
| `discoverDepth` | 否 | `5` | autoDiscover 时的最大递归深度 |

### 5.4 环境变量

| 变量 | 用途 | 必需 |
|------|------|------|
| `NUGET_API_KEY` | NuGet.org API Key | 发布时必需 |
| `OBFUSCAR_PATH` | obfuscar.console 路径 | 混淆时可选（默认从 PATH 查找） |

## 6. 完整流水线步骤说明

`zl-pipeline publish <version>` 按顺序执行以下步骤：

| 步骤 | 名称 | 说明 | 超时 | retry |
|------|------|------|------|-------|
| 1 | **build** | `dotnet build -c Release` 编译项目 | 300s | 1 |
| 2 | **pack** | `dotnet pack -c Release -p:PackageVersion=<version>` 打包 NuGet 到 artifacts 目录 | 600s | 2 |
| 3 | **fix_nuspec** | 解压 nupkg → 修正 nuspec 中 ZL 依赖版本 → 重建 nupkg | 120s | 1 |
| 4 | **publish_deps** | `dotnet publish -o` 发布依赖集到 obfuscated/<project>/publish/（为混淆准备） | 300s | 1 |
| 5 | **obfuscate** | 调用 obfuscar.console 对 DLL 进行混淆 | 300s | 1 |
| 6 | **replace_nupkg** | 调用 `replace-nupkg-dll.py` 将混淆后 DLL 塞回 nupkg | 120s | 1 |
| 7 | **api_compare** | 调用 `api-compare.py` 对比混淆前后公共 API 完整性 | 120s | 1 |
| 8 | **push** | `dotnet nuget push` 推送到 NuGet.org | 120s | 1 |

### 步骤 1: build — 编译

执行 `dotnet build -c Release`。自动设置 `MSBUILDDISABLENODEREUSE=1` 防止 MSBuild 节点崩溃（MSB4166）。

### 步骤 2: pack — NuGet 打包

执行 `dotnet pack -c Release -p:PackageVersion=<version>`，生成的 .nupkg 放在 `artifacts/` 目录。

**注意**：`-p:PackageVersion` 只覆盖包自身版本，不覆盖依赖版本。依赖版本会在步骤 3 修复。

### 步骤 3: fix_nuspec — 修复 nuspec 依赖版本

**问题**：`dotnet pack -p:PackageVersion=X` 只覆盖包自身版本，nuspec 中的 `<dependency>` 仍从 CPM 读取旧版本。

**解决方案**：pack 完成后，自动解压 nupkg → 修正 nuspec 中 ZL 依赖版本 → 重建 nupkg。

**外部包白名单**：`ZL.PFLite`、`ZL.PlcBase`、`ZL.Tag` 不被修复（它们来自独立项目 ZL.PlcBase，版本独立）。

### 步骤 4: publish_deps — 发布依赖集

为混淆准备完整的依赖集。执行 `dotnet publish -c Release -o obfuscated/<project>/publish/`，将项目及其所有依赖 DLL 发布到临时目录。

**原因**：Obfuscar 需要解析所有依赖的类型信息，只传主 DLL 会报错 `Could not load file or assembly ...`。

### 步骤 5: obfuscate — Obfuscar 混淆

对标记 `obfuscate: true` 的项目执行混淆。

**默认混淆配置**：
```xml
<Obfuscator>
  <Var name="KeepPublicApi" value="true" />
  <Var name="HidePrivateApi" value="true" />
  <Var name="UseUnicodeNames" value="true" />
  <Module file="$(InPath)/<name>.dll" />
</Obfuscator>
```

- `KeepPublicApi=true`：保留所有 public 类型和方法名，确保 NuGet 使用者不受影响
- `HidePrivateApi=true`：混淆 private/internal 成员
- `UseUnicodeNames=true`：使用 Unicode 编码混淆名，增加逆向难度

**自定义配置**：在 `pipeline.json` 中指定 `obfuscarConfig` 字段。

### 步骤 6: replace_nupkg — 替换 nupkg 中 DLL

用混淆后的 DLL 替换 `artifacts/<name>.<version>.nupkg` 中的原始 DLL。

**⚠️ 关键实现细节**：使用 read/write 模式直接替换 zip 内文件，**不能使用 append 模式**——append 会导致 nupkg 中出现重复文件，推送时返回 400 Bad Request。

### 步骤 7: api_compare — API 完整性对比

对比混淆前后的公共 API：
1. 提取原始 DLL（来自 nupkg）的所有 public 类型
2. 提取混淆后 DLL 的所有 public 类型
3. 对比：类型名、方法名（public 部分不改变）
4. 100% 匹配 → 通过

如果 API 不一致，视为失败。

### 步骤 8: push — 推送到 NuGet.org

对所有 `artifacts/*.<version>.nupkg` 执行 `dotnet nuget push --skip-duplicate`。

**⚠️ 注意**：`--skip-duplicate` 在 NuGet.org 上有效，但在本地 feed 上无效。推送需要 `NUGET_API_KEY` 环境变量。

## 7. 常见错误与排查

### 7.1 `配置文件不存在: pipeline.json`

**原因**：当前目录不存在 `pipeline.json`。

**解决**：
```bash
zl-pipeline init
# 然后检查生成的 pipeline.json
```

### 7.2 `NUGET_API_KEY 未设置`

**原因**：发布命令需要 API Key。

**解决**：
```bash
export NUGET_API_KEY=your-api-key
```

### 7.3 `obfuscar.console 未安装`

**原因**：混淆步骤需要 obfuscar.console 工具。

**解决**：
```bash
dotnet tool install -g Obfuscar.GlobalTool
```

如果不需要混淆，可在 pipeline.json 中设置 `"obfuscate": false`。

### 7.4 `nupkg 不存在`

**原因**：pack 步骤失败或版本号不匹配。

**解决**：
```bash
# 检查 pack 日志
ls -la artifacts/

# 检查版本号
zl-pipeline plan 1.0.0
```

### 7.5 `NuGet 推送 400 Bad Request`

**原因**：nupkg 中存在重复文件（通常由 append 模式导致）。

**解决**：重新运行 `zl-pipeline publish`，确保使用最新的 zl_pipeline 模块。

### 7.6 `API 对比失败`

**原因**：混淆配置不正确，Obfuscar 混淆了 public API。

**解决**：
- 检查 obfuscar.xml 中 `KeepPublicApi=true`
- 检查 obfuscar.xml 中 `HidePrivateApi=true`
- 查看 Mapping.txt 了解混淆详情

### 7.7 `MSB4166: 子节点崩溃`

**原因**：MSBuild 默认缓存子节点进程，顺序编译多个项目时资源耗尽。

**解决**：zl_pipeline 已自动设置 `MSBUILDDISABLENODEREUSE=1`，无需手动配置。

### 7.8 `PackageId 与项目名不一致`

**原因**：某些项目（如 `ZL.Iot.Runner`）的 PackageId 不等于项目文件名。

**解决**：zl_pipeline 已自动检测 csproj 中的 `<PackageId>`，无需手动干预。

### 7.9 NuGet CDN 传播延迟

**现象**：推送成功后立即验证可能失败（下载的是 HTML 页面而非 ZIP）。

**原因**：NuGet.org CDN 传播延迟，通常 1-5 分钟。

**解决**：等待数分钟后重试 `zl-pipeline check`，或清除本地缓存：
```bash
dotnet nuget locals http-cache -c
```

## 8. 注意事项与最佳实践

### 8.1 发布流程

```bash
# 1. 先 dry-run 验证（不推送）
zl-pipeline verify 1.0.0

# 2. 确认验证通过后正式发布
export NUGET_API_KEY=your-api-key
zl-pipeline publish 1.0.0

# 3. 发布后检查
zl-pipeline check ZL.Watchdog 1.0.0
```

### 8.2 Pipeline 选项说明

| 选项 | 说明 | 使用场景 |
|------|------|---------|
| `--only A B` | 仅处理 A、B 两个项目 | 只发布部分项目 |
| `--from-step obfuscate` | 从混淆步骤开始 | 跳过前面步骤，直接混淆 |
| `--skip-build` | 跳过编译 | 编译已通过，只需要 pack+push |
| `--resume` | 断点续跑 | 上次失败后恢复执行 |

### 8.3 混淆相关注意事项

1. **只标记需要混淆的项目**：Interface 和 Scripting 项目通常设为 `obfuscate: false`
2. **Obfuscar 需要完整依赖集**：publish_deps 步骤会自动准备
3. **混淆配置必须使用变量**：`<Module file="$(InPath)/MyLibrary.dll" />`，不要写死路径
4. **KeepPublicApi 必须为 true**：否则 NuGet 使用者将无法调用 public API

### 8.4 CPM 管理注意事项

1. **所有 .csproj 中不带 Version**：CPM 覆盖范围内的 `PackageReference` 不得含 `Version` 属性
2. **第三方包也入 CPM**：只要启用了 CPM，所有 `PackageReference` 都必须在 CPM 中声明版本
3. **不在 CPM 范围内的项目**：保留 `Version` 属性是正常的

### 8.5 版本管理注意事项

1. **CPM 版本必须在 pack 之前更新**：这是最高优先级原则
2. **不要手动修改 csproj 中的 `<Version>` 字段**：CLI 通过 `-p:PackageVersion` 传入版本号
3. **独立发版时使用 `align-versions`**：自动对齐各 ZL 包到各自最新版本
4. **NuGet.org 已发布版本不可删除**：只能发布新版本替代

### 8.6 pipeline.json 必须纳入版本控制

`pipeline.json` 应提交到 Git 仓库，原因：
- CI/CD 可以读取它
- 团队成员使用一致配置
- 发布历史可追溯

## 附录 B：架构设计 — StepRunner 注册表

zl_pipeline 使用 **StepRunner 注册表** 模式编排流水线步骤：

```python
# 注册步骤
@register_step("build")
def step_build(ctx: PipelineContext, project: dict) -> StepResult:
    ...

@register_step("pack")
def step_pack(ctx: PipelineContext, project: dict) -> StepResult:
    ...
```

所有步骤通过 `@register_step()` 装饰器注册到 `STEP_REGISTRY` 列表，执行顺序即注册顺序。

### 执行引擎

1. **Plan**：生成执行计划（不执行），遍历所有项目 × 所有注册步骤
2. **Run**：编排所有项目的步骤执行
   - 非 resume 模式下任一步骤失败 → 停止后续步骤（abort）
   - resume 模式从 `StateStore` 恢复已通过的步骤
   - 日志输出到 `artifacts/logs/<version>/<step>-<project>.log`
3. **Report**：打印彩色汇总报告

### 状态持久化（断点续传）

- 状态文件：`artifacts/.pipeline-state/<version>/<project>/<step>.json`
- 支持原子写入（先写 .tmp 再 rename）
- resume 模式读取状态文件，跳过已通过的步骤
