结论先说：当前 `/Users/dingyuwang/0-X/deploy/tools` 里的 CLI
    不是行业最佳实践形态，但它是一个“从踩坑中快速堆出来、能解决部分发布痛点的过渡型工具”。
    它能覆盖 build → pack → obfuscate → replace DLL → api compare → push → sync consumers
    这条主链路，但目前结构和验证方式会放大你说的三个问题：容易混乱、编译时间太长、一个错误排查很久。

    2. 是否是最佳实践？

    不是。

    它有很多实用补丁，但还没有达到“发布工具链最佳实践”的标准。

    当前优点

    这些地方是有价值的：

     1. 把发布流程集中到一个 CLI
        - 比散落在多个手动命令里好。
        - 文档中明确了完整链路：

     1      build → pack → publish deps → obfuscate → replace-dll → api-compare → verify-nuget → push

     2. 对历史故障有针对性补丁
        - MSBUILDDISABLENODEREUSE=1
        - retry
        - timeout
        - --stop-on-error
        - dry-run
        - nuspec 依赖版本修复
        - Obfuscar 后 DLL 替换
        - API 对比

     3. 有一定的工程意识
        - 配置文件 pipeline.json
        - schema 文件
        - 文档较完整
        - 有本地发布脚本和下游同步脚本

    这些说明它不是“随便写的脚本”，而是已经经历过真实项目故障后演化出来的工具。

    ---

    3. 但它现在的核心问题很明显

    问题 A：CLI 单文件过大，职责严重混杂

    zl-pipeline.py 同时承担了：

     - 参数解析
     - 配置加载
     - 项目扫描
     - dotnet build
     - dotnet pack
     - nuspec zip 修改
     - Obfuscar XML 生成
     - nupkg DLL 替换
     - API 对比
     - NuGet push
     - 消费方 CPM 修改
     - git add / commit / push
     - 网络查询 NuGet 最新版本
     - 全局状态统计

    这导致：

     - 一个错误可能出现在任何阶段。
     - 阶段之间没有清晰边界。
     - 很难只测试某一步。
     - 修改一个功能容易影响另一条流程。
     - 新成员阅读成本高。

    最佳实践应该是：

      1 cli/
      2   main.py              # 只负责 argparse/click 命令入口
      3   config.py            # pipeline.json 加载与 schema 校验
      4   runner.py            # 命令执行、日志、超时、重试
      5   steps/
      6     build.py
      7     pack.py
      8     fix_nuspec.py
      9     obfuscate.py
     10     replace_nupkg.py
     11     api_compare.py
     12     push.py
     13     sync_consumers.py
     14   dotnet.py            # dotnet 封装
     15   nuget.py             # NuGet 封装
     16   report.py            # 统一结果报告

    当前不是这个形态。

    ---

    问题 B：流程是线性大流水，缺少“可恢复 / 可跳过 / 可定位”的 step 机制

    现在 publish 基本是从头跑：

     1. build
     2. pack
     3. fix nuspec
     4. publish -o
     5. obfuscate
     6. replace DLL
     7. API compare
     8. mapping stats
     9. push

    如果第 7 步失败，前面大量编译和打包已经花掉时间。下一次通常还要重新跑前面步骤。

    这正是你说的：

     │ 每次一个错误排查时间太长

    根因不是单个命令慢，而是没有 step cache 和 resume。

    最佳实践应该支持：

      1 zl-pipeline run build
      2 zl-pipeline run pack
      3 zl-pipeline run obfuscate
      4 zl-pipeline run replace
      5 zl-pipeline run api-compare
      6
      7 zl-pipeline publish 2.3.1 --from-step api-compare
      8 zl-pipeline publish 2.3.1 --only ZL.IotHub
      9 zl-pipeline publish 2.3.1 --skip-build
     10 zl-pipeline publish 2.3.1 --skip-obfuscate
     11 zl-pipeline publish 2.3.1 --resume

    并且每一步产出结构化状态：

     1 .artifacts/pipeline-state/2.3.1/
     2   build.ok
     3   pack.ok
     4   fix-nuspec.ok
     5   obfuscate.ok
     6   replace.ok
     7   api-compare.failed.json

    当前没有。

    ---

    问题 C：dry-run 语义不可靠

    当前 verify 实际是：

     1 args.dry_run = True
     2 cmd_publish(args)

    但 dry_run 下很多步骤只是“假装成功”或跳过：

     - DLL 不存在时 dry-run 直接跳过。
     - nupkg 不存在时 dry-run 跳过。
     - API 对比缺文件时 dry-run 跳过。
     - push 跳过。

    这意味着：

     1 zl-pipeline verify 1.0.1

    并不等于真实验证完整链路。

    它更像是：

     │ 检查一下流程大概能不能走到对应分支，但不保证产物真实正确。

    最佳实践里应拆分：

     1 zl-pipeline plan 2.3.1        # 只打印会做什么，不执行
     2 zl-pipeline verify 2.3.1      # 真实 build/pack/obfuscate/compare，但不 push
     3 zl-pipeline publish 2.3.1     # verify 通过后 push

    现在 dry-run 和 verify 混在一起，会造成误判。

    ---

    问题 D：存在项目特例硬编码，通用性下降

    在 cmd_publish 中有明显特例：

     1 plcbase_dir = Path(proj_dir).parent / "ZL.PlcBase"
     2 if plcbase_dir.exists():
     3     ...
     4     log(f"  从 ZL.PlcBase 同步: {nupkg.name}")

    这说明工具名义上是通用 CLI，但实际混入了 ZL.PlcBase / iot-sdk 的业务假设。

    这会造成：

     - 其他项目使用时行为不可预测。
     - 维护者要知道隐藏规则。
     - 配置文件不是唯一事实来源。
     - “为什么这里会同步别的项目包？”需要读源码才知道。

    最佳实践是把这类关系放进 pipeline.json：

     1 {
     2   "upstreamPackages": [
     3     {
     4       "name": "ZL.PlcBase",
     5       "path": "../ZL.PlcBase/artifacts",
     6       "versionPolicy": "same-version"
     7     }
     8   ]
     9 }

    而不是写死在 Python 里。

    ---

    问题 E：编译时间长的根因没有被系统解决

    当前代码里反复出现：

     - dotnet build
     - dotnet pack
     - dotnet publish
     - Obfuscar
     - API compare 动态创建 C# 项目再 build

    尤其 api-compare.py 每次都会：

     1 tmp_dir = tempfile.mkdtemp(prefix="api-compare-")
     2 dotnet build ApiCompare.csproj
     3 dotnet run --project ApiCompare.csproj

    虽然注释写了：

     │ Cache build artifacts to speed up repeated runs

    但实际代码没有缓存。它每次都创建临时项目并 build。

    这是非常大的耗时来源。

    最佳实践：

     1. 把 API compare 编译成固定工具。
     2. 缓存在：

     1    ~/.cache/zl-pipeline/api-compare/
     3. 或者直接把 C# API compare 做成独立 .NET tool。
     4. 只在源码变更时重新 build。

    另外当前 publish 中先 build 再 pack，dotnet pack 默认可能再次 build。虽然有时 pack 会复用
    build，但最好显式控制：

     1 dotnet build ...
     2 dotnet pack --no-build ...

    如果需要 dotnet publish，也应明确是否可以：

     1 dotnet publish --no-build

    目前没有建立“只编一次”的机制。

    ---

    问题 F：错误报告不够结构化

    当前错误多是：

     1 fail("xxx FAILED")
     2 log(result.stdout + result.stderr)

    全局 PASS / FAIL 只是计数。

    缺少：

     - 当前失败步骤。
     - 项目名。
     - 命令。
     - cwd。
     - exit code。
     - stderr tail。
     - 建议修复。
     - 产物路径。
     - 是否可 resume。
     - 结构化 JSON 日志。

    所以一个错误要排查很久。

    最佳实践输出应该类似：

      1 [FAIL] step=pack project=ZL.IotHub
      2 Command:
      3   dotnet pack src/ZL.IotHub.csproj -c Release --no-build -o artifacts ...
      4
      5 ExitCode: 1
      6 Duration: 48.2s
      7 Log:
      8   artifacts/logs/2026-06-13/pack-ZL.IotHub.log
      9
     10 Likely causes:
     11   1. PackageVersion 与 CPM 不一致
     12   2. build 产物不存在，尝试先运行: zl-pipeline run build --project ZL.IotHub
     13
     14 Resume:
     15   zl-pipeline publish 2.3.1 --from-step pack --only ZL.IotHub

    当前没有这种诊断能力。

    ---

    问题 G：配置 schema 没有真正成为强约束

    存在：

     1 ZL.Pipeline.Cli/schemas/pipeline-schema.json

    但 load_config() 只是：

     1 cfg = json.load(f)
     2 for k, v in DEFAULT_CONFIG.items():
     3     cfg.setdefault(k, v)

    没有看到实际 JSON schema validation。

    这会导致：

     - 字段拼错也继续跑。
     - 类型错也继续跑。
     - 缺少字段可能在很深步骤才报错。
     - 错误定位晚。

    最佳实践：

     1 zl-pipeline config validate

    并在所有命令启动前强校验：

     1 pipeline.json invalid:
     2   projects[2].csproj: file does not exist
     3   consumers[0].path: not absolute or not found
     4   publishTimeout: must be >= 30

    当前配置错误可能到 build / pack 阶段才暴露。

    不过这不是最高优先级。
    真正优先的问题是架构拆分、可恢复、缓存、验证语义。

    5. 最严重的几个缺陷排序

    P0：api-compare.py 结果语义有问题

    C# 程序里如果 API 有差异：

     1 Console.WriteLine("\n[WARNING] Some public API differences detected!");
     2 return 0;

    Python 外层也：

     1 return 0

    这意味着 API 差异不会让流水线失败。

    如果“混淆后公开 API 必须完全保持”是硬要求，这就是严重缺陷。

    应该改成：

     - API 缺失：exit 1。
     - 新增可选是否失败。
     - method mismatch：exit 1。
     - 允许通过配置设置：

     1   "apiCompare": {
     2     "failOnMissing": true,
     3     "failOnAdded": false,
     4     "failOnMethodMismatch": true
     5   }

    ---

    P0：verify / dry-run 会给出虚假的安全感

    现在 verify 是 dry-run，不一定真实产生产物，不一定真实对比。

    这很危险。

    应该分成：

     - plan：不执行，仅展示。
     - verify：真实执行到 API compare，但不 push。
     - publish：先 verify，再 push。
     - --dry-run：只用于展示命令，不应标记 pass。

    ---

    P0：缺少 --only / --from-step / --resume

    这是排查慢的直接原因。

    最小应加：

     1 zl-pipeline publish 2.3.1 --only ZL.IotHub
     2 zl-pipeline publish 2.3.1 --from-step obfuscate
     3 zl-pipeline publish 2.3.1 --resume

    ---

    P1：API compare 每次临时 build，浪费时间

    应该缓存或独立工具化。

    ---

    P1：配置未强校验

    schema 文件存在但没有接入，价值没发挥。

    ---

    P1：项目特例硬编码

    ZL.PlcBase 同步逻辑应移入配置。

    ---

    P1：全局 PASS / FAIL 状态不可控

    全局变量不适合复杂流水线。应该有：

     1 @dataclass
     2 class StepResult:
     3     step: str
     4     project: str
     5     ok: bool
     6     duration: float
     7     command: list[str]
     8     log_file: Path
     9     error_summary: str | None

    6. 行业最佳实践对照

    发布流水线 CLI 最佳实践

    应该具备：

     1. 单一入口
        - 不要多个 shell 脚本各做一套发布逻辑。

     2. 声明式配置
        - 项目、依赖、上游包、下游消费者都在配置里。
        - 源码里不硬编码业务项目名。

     3. 任务 DAG
        - build、pack、obfuscate、replace、verify、push 是明确 step。
        - step 有输入、输出、缓存、失败状态。

     4. 增量与缓存
        - --no-build
        - API compare 工具缓存。
        - 不变项目跳过。
        - 已成功 step 跳过。

     5. 失败可恢复
        - --resume
        - --from-step
        - --only

     6. 结构化日志
        - 每项目每步骤单独 log。
        - 最终 summary 指向 log 文件。

     7. 真实验证
        - verify 必须真实构建和检查产物。
        - dry-run 不能等同 verify。

     8. schema 校验
        - 启动即失败，不要跑到半路才失败。

     9. 测试
        - 配置解析测试。
        - nupkg 替换测试。
        - nuspec 版本修复测试。
        - step runner 测试。
        - API compare 行为测试。

     10. 安全发布
         - push 前确认产物。
         - 不自动 git push，除非明确配置且有保护。
         - secret 不进入日志。

    当前 CLI 大概只满足 1、2、7、10 的一部分。

    ---

    7. 建议改造路线

    第一阶段：止血，不大重构

    目标：马上减少排错时间。

    建议先做：

     1. 修正 api-compare.py：
        - API mismatch 返回非 0。
        - Python 外层返回真实 exit code。

     2. 区分 plan / verify / publish：
        - verify 真实执行但不 push。
        - dry-run 不再标记 pass。

     3. 增加：

     1    --only <project>
     2    --from-step <step>
     3    --skip-build

     4. 每个 step 写独立日志：

     1    artifacts/logs/<version>/<step>-<project>.log

     5. 把失败报告改成：
        - step
        - project
        - command
        - exit code
        - log file
        - stderr tail

    这是最小收益最大的一组。

    ---

    第二阶段：结构化重构

    目标：降低混乱。

    拆 zl-pipeline.py：

      1 ZL.Pipeline.Cli/
      2   zl_pipeline/
      3     __init__.py
      4     main.py
      5     config.py
      6     runner.py
      7     report.py
      8     dotnet.py
      9     nuget.py
     10     steps/
     11       build.py
     12       pack.py
     13       fix_nuspec.py
     14       obfuscate.py
     15       replace_nupkg.py
     16       api_compare.py
     17       push.py
     18       sync_consumers.py

    保留旧入口：

     1 zl-pipeline.py

    只做兼容转发。

    ---

    第三阶段：性能优化

    目标：缩短总耗时。

     1. build 一次，后面使用：

     1    dotnet pack --no-build
     2    dotnet publish --no-build

     2. API compare 工具缓存。

     3. 对独立项目并行执行：
        - build 可按 solution 一次性 build。
