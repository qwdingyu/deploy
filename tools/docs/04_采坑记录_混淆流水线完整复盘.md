## 二、踩坑详细记录

### 坑 #1：CI 独立 job 无共享 bin/ 目录

**现象**：Obfuscar 在独立 job 中找不到 `bin/Release/net8.0/*.dll`

**根因**：GitHub Actions 每个 job 是独立 runner，不共享文件系统

**修复**：合并为单 job，所有 step 在同一 runner 执行

**教训**：CI 流水线设计时就要考虑文件共享问题

---

### 坑 #2：heredoc 变量替换问题

**现象**：Obfuscar XML 中 `${PFLITE_IN}` 未被替换，InPath 值为空

**根因**：
1. `'EOF'`（带引号）阻止所有变量替换
2. `envsubst` 在 GitHub Actions ubuntu-latest 中行为不稳定
3. `$()` 命令替换与 Obfuscar 的 `$(InPath)` 语法冲突

**迭代过程**：
| 迭代 | 方案 | 结果 |
|------|------|------|
| 1 | heredoc 直接替换 | ❌ `$(InPath)` 被 bash 解析 |
| 2 | `'EOF'` + envsubst | ❌ InPath 值为空 |
| 3 | sed + 模板文件 | ❌ 仍缺依赖 DLL |
| 4 | `printf '%s'` 格式化 | ✅ 稳定可靠 |

**最终方案**：
```bash
printf '<Obfuscator>\n  <Var name="InPath" value="%s" />\n  ...  <Module file="$(InPath)/ZL.PFLite.dll" />\n</Obfuscator>\n' "$PDIR" "$ODIR" > obfuscar.xml
```

**教训**：
- heredoc 在 GitHub Actions 中不可靠
- `$(InPath)` 是 bash 和 Obfuscar 的语法冲突
- `printf` 是最稳定的 XML 生成方式

---

### 坑 #3：bin/Release/net8.0 缺少依赖 DLL

**现象**：Obfuscar 报错找不到 HslCommunication.dll、S7.Net.dll 等 30+ 依赖

**根因**：`dotnet build` 只输出项目自身 DLL，不复制第三方依赖

**对比**：
```bash
# build 输出（缺少依赖）
ls ZL.PlcBase/bin/Release/net8.0/
→ ZL.PlcBase.dll, ZL.PlcBase.deps.json, ZL.PlcBase.pdb, ZL.PlcBase.xml
→ 缺少：HslCommunication.dll, S7.Net.dll, Newtonsoft.Json.dll, ...

# publish 输出（完整依赖）
dotnet publish -c Release -f net8.0 -o publish-obs/ZL.PlcBase
→ ZL.PlcBase.dll + 30+ 依赖 DLL
```

**修复**：用 `dotnet publish -o` 替代 `dotnet build`

**教训**：
- `dotnet build` ≠ `dotnet publish`
- Obfuscar 需要完整依赖集才能解析
- publish -o 比 build 慢 3 倍（ZL.PlcBase: 21s vs 68.8s）

---

### 坑 #4：pack 后再混淆 vs 混淆后再 pack

**问题**：先混淆再 pack 会破坏 bin/ 目录的多 TFM 结构

**方案对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| 混淆 → pack | 直接生成混淆包 | 破坏 bin/ 多 TFM，pack 可能失败 |
| **pack → 混淆 → 替换 nupkg** | **不破坏 bin/，安全** | **需要额外替换步骤** |

**最终方案**：pack 先行（用干净 bin/），混淆后用 Python 脚本替换 nupkg 中的 DLL

---

### 坑 #5：Python zipfile 'a' 模式追加而非替换

**现象**：NuGet 推送 400 Bad Request — "duplicated files"

**根因**：`zipfile.ZipFile(path, 'a')` 是**追加**模式，不是替换模式

**代码对比**：
```python
# ❌ 第1版：append 模式（追加条目，不删除旧的）
with zipfile.ZipFile(nupkg, 'a') as zf:
    zf.write(new_dll, 'lib/net8.0/ZL.PlcBase.dll')
# 结果：nupkg 中有 2 个 lib/net8.0/ZL.PlcBase.dll

# ✅ 第3版：read-all → write-new 模式
with zipfile.ZipFile(tmp, 'r') as zin:
    with zipfile.ZipFile(out, 'w', zipfile.ZIP_DEFLATED) as zout:
        for info in zin.infolist():
            if info.filename == target:
                zout.write(new_dll, target)  # 替换
            else:
                zout.writestr(info, zin.read(info.filename))  # 原样复制
# 结果：nupkg 中只有 1 个 lib/net8.0/ZL.PlcBase.dll
```

**教训**：Python zipfile 的 'a' 模式行为与直觉不同

---

### 坑 #6：xml.etree.ElementTree 序列化引入 ns0: 前缀

**现象**：NuGet 推送 400 Bad Request — "unexpected error"

**根因**：Python `xml.etree.ElementTree.tostring()` 序列化时会为默认命名空间添加 `ns0:` 前缀

**代码对比**：
```xml
<!-- 原始 nuspec（正确） -->
<package xmlns="http://schemas.microsoft.com/packaging/2013/05/nuspec.xsd">
  <metadata>...</metadata>
</package>

<!-- ET 序列化后（错误） -->
<ns0:package xmlns:ns0="http://schemas.microsoft.com/packaging/2013/05/nuspec.xsd">
  <ns0:metadata>...</ns0:metadata>
</ns0:package>
```

**修复**：第3版脚本完全不碰 nuspec XML，只替换 DLL 二进制文件

**教训**：
- Python ET 不适合修改已有命名空间的 XML
- 即使只更新 hash，ET 序列化也会破坏 XML 结构
- **最优方案：完全不碰 nuspec**

---

### 坑 #7：tag push 时 inputs 不存在

**现象**：tag push 触发 CI，但混淆 step 被跳过

**根因**：`if: inputs.obfuscate == true` 在 tag push 时 `inputs` 为 null

**修复**：
```yaml
# ❌ 旧：tag push 时永远为 false
if: inputs.obfuscate == true

# ✅ 新：tag push 默认混淆，workflow_dispatch 可控制
if: github.event_name != 'workflow_dispatch' || inputs.obfuscate == true
```

**教训**：GitHub Actions 的 `inputs` 只在 `workflow_dispatch` 时存在

---

### 坑 #8：临时脚本不可复用

**现象**：api-compare.py 在 /tmp/ 创建，每次都要重写

**根因**：没有"脚本固化"意识，临时命令散落多轮对话

**修复**：
1. 固化 api-compare.py 到 scripts/
2. 固化 replace-nupkg-dll.py 到 scripts/
3. 创建 release-verify.sh 编排完整流水线
4. 同步到 ZL.PlcSimulator

**教训**：**所有过程必须固化为脚本**，临时命令的代价 = 编写时间 × 使用次数 + 调试时间 × 失败次数

## 三、最优解决方案

### 3.1 完整流水线架构

```
┌─────────────────────────────────────────────────────────────┐
│  scripts/release-verify.sh <version> [--dry-run]            │
│                                                             │
│  Step 0: 环境检查                                           │
│  Step 1: dotnet build -c Release                            │
│  Step 2: dotnet pack -c Release --no-build                  │
│  Step 3: dotnet publish -f net8.0 -o publish-obs/<Proj>/    │
│  Step 4: obfuscar.console obfuscar.<Proj>.xml               │
│  Step 5: python3 scripts/replace-nupkg-dll.py               │
│  Step 6: python3 scripts/api-compare.py                     │
│  Step 7: 混淆强度统计                                       │
│  Step 8: 运行时验证                                         │
│  Step 9: 报告                                               │
└─────────────────────────────────────────────────────────────┘
         │
         ▼ 全部通过
dotnet nuget push artifacts/packages/*.nupkg -k $NUGET_API_KEY
```

### 3.2 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 依赖获取 | `dotnet publish -o` | 输出完整依赖集，Obfuscar 可直接使用 |
| XML 生成 | `printf` 格式化 | 避免 heredoc 变量替换和 `$()` 冲突 |
| nupkg 修改 | read-all → write-new | 避免 zipfile 'a' 模式追加问题 |
| nuspec 处理 | 完全不碰 | 避免 ET 序列化命名空间污染 |
| API 验证 | 反射对比 + 运行时测试 | 确保混淆不破坏公共 API |
| 脚本管理 | 固化到 scripts/ | 可复用、可版本控制、可审计 |

### 3.3 质量门禁

| 门禁 | 检查内容 | 工具 |
|------|---------|------|
| G0 | 环境完整性 | dotnet/obfuscar/python3/脚本存在性 |
| G1 | 构建通过 | `dotnet build` 0 错误 |
| G2 | 打包成功 | nupkg 文件存在 |
| G3 | 依赖完整 | publish -o 输出完整 |
| G4 | 混淆成功 | Obfuscar "Completed" + DLL 存在 |
